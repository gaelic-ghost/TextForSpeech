# TextForSpeech v0.19.0

## What changed

- moved path context fully onto `TextForSpeech.RequestContext`
- removed caller-provided text-format hints from mixed-text normalization
- removed mixed-text nested-source hints so fenced and inline code now use generic embedded-code speech unless the whole input uses the source lane
- removed `TextForSpeech.InputContext` and the public `withContext:` normalization arguments
- added `swift-markdown` and SwiftSoup dependencies for parser-backed Markdown and HTML structure detection
- replaced custom Markdown handling for fenced code, inline code, links, and priority labels with `swift-markdown` traversal and source-range replacement
- added an internal `AttributedString` semantic-run surface for platform and developer token annotations
- updated README, ROADMAP, and maintainer docs to reflect the current request-context, parser-backed detection, semantic-run, and style-review direction

## Breaking changes

- `TextForSpeech.InputContext` has been removed
- `TextForSpeech.Normalize.text(..., withContext:)` has been removed
- `TextForSpeech.Normalize.source(..., withContext:)` has been removed
- `runtime.normalize.text(..., withContext:)` has been removed
- `runtime.normalize.source(..., withContext:)` has been removed
- mixed-text normalization no longer accepts request-wide text-format or nested-source hints

## Migration notes

- pass `cwd`, `repoRoot`, and request metadata with `TextForSpeech.RequestContext`
- call `TextForSpeech.Normalize.text(...)` for mixed prose, Markdown, HTML, logs, CLI output, or agent text
- call `TextForSpeech.Normalize.source(..., as:)` only when the whole input is source text and the caller already knows the source format
- remove `withContext:` arguments at call sites
- remove `InputContext(nestedSourceFormat:)` usage; embedded code in mixed text now uses generic code speech
